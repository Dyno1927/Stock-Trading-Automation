# Support

# Stock Trading Automation (STA)

Version: Draft

Status: Active

---

# Purpose

This document explains how to request help, report problems, ask questions, and suggest improvements for the Stock Trading Automation (STA) project.

Providing complete and accurate information helps resolve issues more efficiently.

---

# Before Asking for Help

Before opening an Issue or requesting support:

* Read the README.
* Review the project documentation.
* Check the ROADMAP.
* Review existing Issues.
* Search previous Discussions.
* Verify that the issue still exists on the latest version.

Many common questions may already be answered.

---

# Reporting Bugs

When reporting a bug, include:

## Summary

A short description of the problem.

---

## Expected Behavior

Describe what should have happened.

---

## Actual Behavior

Describe what actually happened.

---

## Steps to Reproduce

Provide clear, repeatable steps.

Example:

1. Start STA.
2. Load market data.
3. Execute strategy.
4. Observe unexpected behavior.

---

## Environment

Include:

* Operating System
* Python Version
* Node.js Version
* Git Commit or Release Version
* Database (if applicable)

---

## Logs

Include relevant logs whenever possible.

Remove any sensitive information before sharing.

---

# Feature Requests

Feature requests should explain:

* The problem being solved.
* Why the feature is useful.
* Possible implementation ideas.
* Alternative approaches.
* Potential drawbacks.

Features that align with the project's architecture are significantly more likely to be accepted.

---

# Documentation Issues

Documentation improvements are always welcome.

Examples include:

* Missing explanations
* Typographical errors
* Outdated information
* Incorrect examples
* Broken links

Documentation quality is considered as important as code quality.

---

# Security Issues

Do **not** report security vulnerabilities through public GitHub Issues.

Instead:

* Contact the maintainers privately.
* Provide a detailed description.
* Include reproduction steps if appropriate.
* Allow reasonable time for investigation before public disclosure.

Responsible disclosure helps protect users.

---

# Questions

General questions should preferably use:

* GitHub Discussions
* Project Discussions
* Future community channels

Questions that improve the documentation may later be incorporated into the project.

---

# Suggestions

Suggestions are encouraged.

Good suggestions usually include:

* The problem
* The proposed improvement
* Expected benefits
* Possible trade-offs

Constructive feedback helps improve the project.

---

# Response Priorities

Support requests are generally handled in the following order:

1. Security issues
2. Critical bugs
3. Data integrity issues
4. Trading-related issues
5. Documentation improvements
6. Feature requests
7. General questions

Priority may change depending on project needs.

---

# Project Status

STA is currently under active development.

Some planned functionality may not yet be implemented.

The ROADMAP provides the most accurate overview of upcoming features.

---

# Support Philosophy

Support is provided on a best-effort basis during the early stages of development.

As the project matures, support processes and response times may evolve.

---

# Guiding Principle

A well-written bug report or feature request saves time for everyone.

The more information provided, the easier it becomes to investigate, reproduce, and resolve an issue.
